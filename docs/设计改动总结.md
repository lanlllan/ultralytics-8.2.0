本文档基于本仓库 `.git` 历史整理，目标不是复述每一次提交，而是说明本项目围绕“注意力机制改进”和“损失函数改进”具体做了哪些工作，以及这些改动在代码层面是如何落地的。

从提交历史看，当前仓库的定制开发大致分为四个阶段：

1. 初始导入 Ultralytics 8.2.0 基线代码。
2. 引入注意力模块，并为 YOLOv8-seg 设计多组结构改进方案。

4. 针对具体任务目标重构 Fitness 指标，使模型选择不再只依赖默认 `mAP50-95` 权重。

## 2. 注意力机制改进

### 2.1 做了什么

注意力方向的核心目标，是在不完全推翻 YOLOv8-seg 主体结构的前提下，引入轻量注意力模块，提升特征表达能力，尤其关注分割任务中多尺度特征融合阶段的表现。

本项目新增了 4 类注意力模块的代码注册能力：

- `CBAM`
- `SEAttention`
- `ECAAttention`
- `SimAM`
- `CoordAtt`

其中真正用于实验配置的主要是：

- `CBAM`
- `SimAM`
- `CoordAtt`

相关实现文件：

- `ultralytics/nn/modules/conv.py`
- `ultralytics/nn/modules/__init__.py`
- `ultralytics/nn/tasks.py`
- `ultralytics/cfg/models/v8/*.yaml`

### 2.2 是如何实现的

实现分成三步。

第一步，在模块层增加注意力算子。

在 `ultralytics/nn/modules/conv.py` 中新增了 `SEAttention`、`ECAAttention`、`SimAM`、`CoordAtt` 类。这样做的好处是：

- 保持与原始 Ultralytics 模块组织方式一致；
- 新模块可以像 `Conv`、`C2f`、`SPPF` 一样被模型解析器直接识别；
- 便于后续通过 YAML 搭积木，而不是手工改网络前向过程。

第二步，在模块导出层和模型解析层注册新模块。

在 `ultralytics/nn/modules/__init__.py` 中将这些模块加入 `__all__` 导出列表；在 `ultralytics/nn/tasks.py` 的 `parse_model()` 中加入：

- 对这些注意力模块的导入；
- 对输入输出通道 `c1/c2` 的解析逻辑；
- 让注意力模块按“输入输出通道数相同”的方式接入现有网络图。

这一步是关键。因为 YOLOv8 的模型并不是硬编码搭建，而是通过 YAML 配置交给 `parse_model()` 动态解析。如果不在 `parse_model()` 中注册，YAML 里即使写了 `CoordAtt` 或 `SimAM`，框架也无法构建模型。

第三步，新增多份分割模型 YAML，用于表达不同插入位置的结构改造方案。

新增配置文件包括：

- `ultralytics/cfg/models/v8/yolov8-seg-ca-a.yaml`
- `ultralytics/cfg/models/v8/yolov8-seg-ca-c.yaml`
- `ultralytics/cfg/models/v8/yolov8-seg-cbam-a.yaml`
- `ultralytics/cfg/models/v8/yolov8-seg-cbam-c.yaml`
- `ultralytics/cfg/models/v8/yolov8-seg-simam-a.yaml`

这里体现了一个很清晰的实验设计思路：不仅比较“哪种注意力更好”，也比较“加在什么位置更有效”。

### 2.3 具体怎么插入的

从 YAML 可看出，本项目主要探索了两种插入位置。

#### 位置 A：Backbone 末尾插入

典型做法是在 `SPPF` 后插入注意力层，例如：

- `SPPF -> CoordAtt`
- `SPPF -> CBAM`
- `SPPF -> SimAM`

其思路是：

- 在主干网络完成高层语义提取后再做一次通道/空间重标定；
- 尽量减少对下游 Neck 主结构的破坏；
- 便于与原版 backbone 做对照。

#### 位置 C：Neck 多尺度融合阶段插入

例如在每个 Neck 的 `C2f` 融合输出后再接一个注意力块：

- P4 融合后加注意力；
- P3 融合后加注意力；
- 下采样回 P4 后再加注意力；
- 下采样回 P5 后再加注意力。

这种设计明显更偏向分割任务，因为：

- 分割更依赖多尺度细节和语义融合；
- Neck 阶段的特征融合质量直接影响 mask 预测；
- 在 Neck 插入可以避免大改 backbone，预训练权重兼容性更好。

从注释看，你在设计中明确考虑了“backbone 完全不变，预训练权重 100% 兼容”的问题，这一点很适合在论文中强调，因为它解释了为什么改动选择插在 Neck 而不是从 backbone 早期就重构。

### 2.4 相关实验工程化

注意力实验不是只停留在模型文件层面，而是配套了专门的实验脚本：

- `cmd/yolov8-seg-attention-train.py`
- `cmd/yolov8-seg-attention-val.py`

这两个脚本的作用分别是：

- 统一启动多组注意力实验，保证训练参数一致，减少手工切换模型带来的误差；
- 统一扫描 `runs/segment/attnv2_*` 的结果，输出可直接用于论文表格的评估结果。

这说明本项目在注意力部分不仅完成了模型改造，也完成了实验流程标准化。

## 3. 损失函数改进

### 3.1 做了什么

损失函数方向包含两个层面：

1. 边界框回归损失改进。
2. 分割掩码损失改进。

边界框回归方面，新增支持：

- `EIoU`
- `SIoU`
- `WIoU`

掩码损失方面，新增支持：

- `Dice`
- `BCE-Dice`

相关实现文件：

- `ultralytics/utils/loss.py`
- `ultralytics/utils/metrics.py`
- `ultralytics/cfg/default.yaml`
- `cmd/yolov8-seg-loss-train.py`
- `cmd/yolov8-seg-loss-val.py`

### 3.2 是如何实现边界框损失改进的

边界框损失改进的核心逻辑分成三层。

第一层，在配置层引入可切换参数。

在 `ultralytics/cfg/default.yaml` 中新增：

- `iou_type: CIoU`

这意味着训练时可以通过参数直接切换使用何种 IoU 损失，而不是为每一种损失单独写一套训练流程。

第二层，在几何指标函数层扩展 `bbox_iou()`。

在 `ultralytics/utils/metrics.py` 中，原始 `bbox_iou()` 只支持 `IoU/GIoU/DIoU/CIoU`。项目扩展后支持：

- `EIoU`
- `SIoU`

这里的实现要点是：

- `EIoU` 引入了宽度和高度差异惩罚；
- `SIoU` 引入了角度成本、距离成本和形状成本；
- 后续又通过提交 `e25bf5d` 对 `SIoU` 的角度项和距离项进行了修正，说明你不是机械移植公式，而是对实现细节做了调试和修复。

第三层，在训练损失装配层让 `BboxLoss` 接受 `iou_type`。

在 `ultralytics/utils/loss.py` 中：

- `BboxLoss` 新增 `iou_type` 参数；
- `v8DetectionLoss` 在初始化时从超参数中读取 `iou_type`；
- 针对 `WIoU` 单独实现了加权逻辑，而不是简单通过 `bbox_iou()` 复用。

这里体现出一个重要实现特点：

- `EIoU/SIoU` 是作为 `bbox_iou()` 的不同分支来实现；
- `WIoU` 因为需要额外的动态聚焦权重 `wise_scale`，所以在 `BboxLoss.forward()` 里单独计算。

也就是说，你不是把所有新损失都硬塞到同一层，而是根据算法性质拆到了更合适的位置。

### 3.3 是如何实现掩码损失改进的

分割损失改进主要集中在 `ultralytics/utils/loss.py` 的 `v8SegmentationLoss`。

实现过程同样分成三层。

第一层，在默认配置中加入可切换项：

- `seg_loss_type: BCE`

第二层，新增 `dice_loss()` 辅助函数。

这里采用的思路是：

- 先对预测 logits 做 `sigmoid`；
- 再拉平为向量；
- 通过预测与真值的交集和并集构造 Dice 系数；
- 最终以 `1 - Dice` 作为损失。

第三层，在 `single_mask_loss()` 中统一调度不同掩码损失。

你把实例级 mask loss 改造成了三种模式：

- `BCE`：保持原版逻辑；
- `Dice`：只使用 Dice Loss；
- `BCE-Dice`：将 BCE 与 Dice 叠加。

并且这里不是直接对整张原图算损失，而是仍然保留了原版 YOLOv8 的 `crop_mask()` 思路，即：

- 先由 `pred @ proto` 得到实例 mask；
- 再按目标框范围裁剪有效区域；
- 只在目标相关区域内计算掩码损失。

这个实现非常重要，因为它说明你的改进并没有破坏 YOLOv8-seg 原有的 prototype mask 机制，而是在原框架之上替换损失计算项，属于“低侵入式改造”。

### 3.4 训练脚本如何支撑损失消融

`cmd/yolov8-seg-loss-train.py` 是这一部分的实验中枢。它做了几件很关键的工程化工作：

- 将所有损失实验定义为 `EXPERIMENTS` 字典；
- 明确区分 IoU 损失改进组和掩码损失改进组；
- 固定训练参数，确保不同实验之间只有损失函数变化；
- 支持 `resume`，方便长时间实验断点续训；
- 后续又加入 `--box` 参数，用于调节 box loss 增益；
- 支持跳过 baseline，复用前面注意力实验的基线权重。

配套的 `cmd/yolov8-seg-loss-val.py` 则统一完成：

- 自动扫描实验结果；
- 逐个加载最佳权重并验证；
- 导出总表和消融表；
- 输出便于比较的对照结果。

这种脚本化方式使得你的损失函数研究不只是“改了代码”，而是形成了可重复实验流程。

## 4. Fitness 评估体系改进

### 4.1 为什么要改

Ultralytics 默认的 Fitness 更强调：

- `0.1 * mAP50 + 0.9 * mAP50-95`

这适合通用目标检测/分割任务，但不一定适合你的具体应用目标。根据提交说明和脚本注释，你后续实验更关注 OCR 流水线场景下的漏检和高阈值定位质量，因此仅看默认 `mAP50-95` 不够。

所以你新增了一套自定义 Fitness：

- `recall_map75 = 0.2 * Recall + 0.5 * mAP@0.75 + 0.3 * mAP@0.5:0.95`

这个公式的设计意图很明确：

- 提高 Recall 权重，降低漏检风险；
- 强调 `mAP75`，更看重较高 IoU 阈值下的定位/分割质量；
- 保留 `mAP50-95` 作为总体性能约束。

### 4.2 是如何实现的

这个部分的改动非常系统，不是简单在脚本里手算一个分数，而是把新 Fitness 嵌入到了框架内部。

第一步，在 `ultralytics/utils/metrics.py` 中给 `Metric` 增加 `FITNESS_PRESETS`：

- `default`
- `recall_map75`

然后给 `Metric` 新增 `fitness_type` 属性，并改写 `fitness()` 逻辑，使其能够根据预设选择不同指标组合。

第二步，把 `fitness_type` 往各类评估器中传递。

相关修改包括：

- `ultralytics/models/yolo/detect/val.py`
- `ultralytics/models/yolo/segment/val.py`
- `ultralytics/models/yolo/pose/val.py`
- `ultralytics/models/yolo/obb/val.py`
- 后续还扩展到 `ultralytics/models/fastsam/val.py`

这样做的效果是：

- 不管是 detect、segment、pose、obb 还是 FastSAM，验证阶段都能共享统一的 Fitness 体系；
- 新旧指标的对比不再局限于单个脚本内部，而是进入框架级评估流程。

第三步，在实验脚本中显式支持 `fitness_type` 参数。

你把这个能力接到了：

- `cmd/yolov8-seg-attention-train.py`
- `cmd/yolov8-seg-attention-val.py`
- `cmd/yolov8-seg-loss-train.py`
- `cmd/yolov8-seg-loss-val.py`
- `cmd/yolov8-seg-fitness-eval.py`

这意味着同一批权重既可以按默认 Fitness 排名，也可以按新 Fitness 排名，从而观察“最优模型是否发生变化”。

### 4.3 这个改动的意义

这部分实际上把你的毕业设计从“模型结构调参”推进到了“任务目标对齐”层面。

也就是说，项目不只是问“哪个模型在通用指标上更高”，而是进一步问：

- 对于我的实际业务目标，什么样的模型才算更优？
- 默认评估标准是否会选错模型？
- 当任务更重视 Recall 和高阈值精度时，评价函数是否也要同步调整？

这一点很适合作为论文中较高层次的创新点来写，因为它体现了“评价标准与应用目标一致性”的思考。

## 6. 可以在论文中重点强调的“如何做”

如果论文要突出“实现过程”而不是只给结果，建议重点写以下几个方面。

### 6.1 基于 YAML + `parse_model()` 实现低侵入式结构改造

注意力机制的引入没有重写 YOLOv8 主干，而是遵循 Ultralytics 原本的模型解析机制：

- 新增模块类；
- 注册到模块导出；
- 注册到 `parse_model()`；
- 通过 YAML 进行结构插入。

这使得结构改动具备：

- 可扩展；
- 易复现；
- 便于做多组对照实验；
- 与预训练权重兼容性较好。

### 6.2 基于统一超参数接口实现可切换损失

损失函数不是分别做成多个分叉版训练器，而是通过：

- `iou_type`
- `seg_loss_type`

两个超参数在同一训练框架中切换。这种实现方式更规范，也更适合做消融实验，因为：

- 训练流程保持一致；
- 非目标变量影响更小；
- 对比更公平。

### 6.3 将新评估标准下沉到框架级而不是脚本级

Fitness 改造最有价值的一点，是它被下沉到了 `Metric` 和各类 `Validator` 内部，而不是停留在“训练完成后我自己写个 Excel 加权算分”。

这样做带来的优势是：

- 训练过程中的 best model 选择可以直接受新 Fitness 影响；
- 不同任务共享统一评估逻辑；
- 新旧指标可以并行对比；
- 代码体系更完整。

### 6.4 为长周期实验建立批量化脚本

训练和验证脚本普遍支持：

- 实验矩阵定义；
- 自动扫描结果；
- 输出 CSV 表格；
- 断点续训；
- 统一命名和统一目录结构。

这意味着你的工作不只是“实现一个新模块”，而是建立了一套可执行的研究流程。

## 8. 总结

综合 `.git` 历史可以看出，本项目的毕业设计工作并不是零散地改几个文件，而是形成了一条比较完整的技术路线：

1. 以 Ultralytics YOLOv8-seg 为基线。
2. 通过注意力模块在 Backbone / Neck 上进行结构增强。
3. 通过 IoU 损失和掩码损失改进优化训练目标。
4. 通过自定义 Fitness 让模型选择标准更贴近具体任务需求。
5. 通过批量训练、批量评估和结果导出脚本实现实验流程工程化。

如果从“实现方法”角度概括，你的主要贡献可以表述为：

- 在 Ultralytics 原有动态组网框架上完成了可配置的注意力结构扩展；
- 在 YOLOv8-seg 原有损失体系上完成了可切换的回归损失和掩码损失扩展；
